Elliptic Curves in the Real World: Cryptography

The group law from Module 2 — "draw a line through two points, reflect the third intersection" — turns out to be the security primitive protecting most of the internet's encrypted traffic. This module shows how the same geometric structure becomes a one-way trapdoor when you move from the real line to a finite field.

From $\mathbb{R}$ to $\mathbb{F}_p$

Take the Weierstrass equation $y^2 = x^3 + ax + b$ and replace $\mathbb{R}$ with the finite field $\mathbb{F}_p = \{0, 1, \ldots, p-1\}$ for a large prime $p$. All arithmetic is modulo $p$: $13 + 5 \equiv 3 \pmod 5$, and division becomes multiplication by the modular inverse ($a^{-1} \equiv a^{p-2} \pmod p$ by Fermat's little theorem).

The curve $E(\mathbb{F}_p)$ is now a finite set of points — the pairs $(x, y)$ with $x, y \in \mathbb{F}_p$ satisfying the equation, plus the point at infinity $\mathcal{O}$. You cannot draw it continuously; it appears as a scatter of discrete dots on a $p \times p$ grid.

The point addition formula is identical to the real case (same algebra, same slope calculation) — just with all arithmetic reduced mod $p$:

$$\lambda = \begin{cases} \dfrac{y_Q - y_P}{x_Q - x_P} \pmod{p} & P \neq Q \\[6pt] \dfrac{3x_P^2 + a}{2y_P} \pmod{p} & P = Q \end{cases}$$
$$x_R = \lambda^2 - x_P - x_Q \pmod{p}, \qquad y_R = \lambda(x_P - x_R) - y_P \pmod{p}$$

The group axioms still hold: addition is associative, $\mathcal{O}$ is the identity, and every point has an inverse $(x, y)^{-1} = (x, p-y)$.

Group Order and Hasse's Theorem

How many points does $E(\mathbb{F}_p)$ have? Hasse's theorem (1933) gives the tight bound:

$$\left|\,\#E(\mathbb{F}_p) - (p + 1)\,\right| \leq 2\sqrt{p}$$

So the group has roughly $p$ elements — close to $p$, but not exactly. The deviation $t = p + 1 - \#E(\mathbb{F}_p)$ is called the trace of Frobenius and encodes deep arithmetic information about the curve.

Scalar Multiplication and the Hard Problem

Fix a public base point $G \in E(\mathbb{F}_p)$, called the generator. Define scalar multiplication:

$$nG = \underbrace{G + G + \cdots + G}_{n \text{ times}}$$

This is computed efficiently in $O(\log n)$ operations using double-and-add — the same idea as fast exponentiation. With $p \approx 2^{256}$, computing $nG$ from $n$ takes microseconds even for $n$ with 256 bits.

The Elliptic Curve Discrete Logarithm Problem (ECDLP) is the reverse: given $G$ and $Q = nG$, find $n$. No polynomial-time algorithm is known. The best classical attack (Pollard's rho) runs in $O(\sqrt{p})$ time — for $p \approx 2^{256}$ that is $2^{128}$ operations, computationally infeasible.

Why is this hard? Unlike ordinary discrete logarithms (where the structure of $\mathbb{Z}/p\mathbb{Z}^*$ allows sub-exponential attacks via index calculus), the group $E(\mathbb{F}_p)$ has no obvious sub-group structure to exploit. The points jump around the grid apparently at random as $n$ increases.

ECDH Key Exchange

The ECDLP hardness enables a secure key exchange over a public channel:

  1. Public parameters: everyone knows the curve $E(\mathbb{F}_p)$ and generator $G$.
  2. Alice picks a secret integer $a$, computes $A = aG$, and publishes $A$.
  3. Bob picks a secret integer $b$, computes $B = bG$, and publishes $B$.
  4. Alice computes $a \cdot B = a(bG) = abG$.
  5. Bob computes $b \cdot A = b(aG) = abG$.
  6. Both arrive at the same shared secret $abG$ — which Eve cannot compute from $A = aG$ and $B = bG$ alone without solving the ECDLP.

This is ECDH (Elliptic Curve Diffie-Hellman). It is used in TLS 1.3, SSH, Signal, and most modern encrypted connections.

Real Curves

Two curves dominate practice:

Curve Equation Prime $p$ Used in
secp256k1 $y^2 = x^3 + 7$ $2^{256} - 2^{32} - 977$ Bitcoin, Ethereum
NIST P-256 $y^2 = x^3 - 3x + b$ $2^{256} - 2^{224} + 2^{192} + 2^{96} - 1$ TLS, HTTPS, passkeys

For both curves, the group order $\#E(\mathbb{F}_p)$ is a 256-bit prime (or very close to one), ensuring the strongest possible resistance to ECDLP attacks.

Why ECC Beats RSA

RSA security rests on the hardness of integer factorization. The best known classical algorithm (General Number Field Sieve) runs in sub-exponential time $O\!\left(\exp\left(c \cdot n^{1/3} (\log n)^{2/3}\right)\right)$ in the key length $n$. ECDLP has no sub-exponential attack — the best is $O(\sqrt{p})$, which is fully exponential in the key length.

The practical consequence:

Security level RSA key size ECC key size
80-bit1024 bits160 bits
128-bit3072 bits256 bits
256-bit15360 bits512 bits

Smaller keys mean faster handshakes, smaller certificates, and less computation — critical for mobile devices and embedded systems.
